# Nautica Production Deployment Guide (Ubuntu 22.04, Nginx, Cloudflare)

This guide is written specifically for the current Nautica codebase in this repository.
It assumes:

- Ubuntu Server 22.04 LTS
- Nginx as the public reverse proxy
- Cloudflare proxy enabled for the domain
- the production server uses a canonical split:
  - frontend served from `/websites/nautica/frontend`
  - backend code run from `/websites/nautica/backend`
  - SQLite database at `DATABASE_PATH`
  - uploaded media at `MEDIA_UPLOAD_DIR`

---

## 1. How this project runs in production

Current production shape for this repo:

- **Frontend**: static build generated by `npm run build`
- **Prerendered public routes**: generated by `npm run prerender` during build
- **API**: Express app from `apps/api/src/server.ts`
- **API runtime**: compiled JavaScript from `server-dist/` via `npm run start:api`
- **Database**: SQLite file
- **Uploads**: local filesystem storage

### Important repo-specific note

The current `build` script emits compiled JavaScript for the API into `server-dist/`.

That means the production path is now:

- build frontend assets into `dist/`
- build the API into `server-dist/`
- run the API through `systemd`
- let Nginx serve `dist/` and proxy `/api`, `/health`, and `/ready`

---

## 2. Recommended server layout

Use a canonical `frontend / backend / shared` layout so uploaded media and the database stay outside deployable code.

```text
/websites/nautica/
├── frontend/
├── backend/
├── shared/
│   ├── db/
│   │   └── nautica.sqlite
│   └── uploads/
└── backups/
```

Use a separate environment file outside the release tree:

```text
/etc/nautica/nautica.env
```

This keeps secrets out of both the frontend and backend trees, while `shared/` remains the only persistent data layer.

---

## 3. DNS and Cloudflare model

Target architecture:

```text
Visitor -> Cloudflare -> Nginx (Ubuntu) ->
  - static frontend from `/websites/nautica/frontend`
  - /api, /health, /ready proxied to Node API on 127.0.0.1:8787
  - /uploads served directly from `/websites/nautica/shared/uploads`
```

Recommended Cloudflare setup:

- DNS record proxied through Cloudflare (orange cloud)
- SSL/TLS mode: **Full (strict)**
- install a valid origin certificate on Nginx
- optionally enable **Always Use HTTPS** after the site works correctly

Give Nginx read access to shared uploads:

```bash
usermod -aG nautica www-data
chmod 750 /websites/nautica/shared
chmod 750 /websites/nautica/shared/uploads
find /websites/nautica/shared/uploads -type d -exec chmod 750 {} \;
find /websites/nautica/shared/uploads -type f -exec chmod 640 {} \;
```

Log out and back in, or restart Nginx later, so the new group membership is picked up.

## 6. Upload the project to the server

From your local machine, upload the ZIP to the server, for example:

```bash
scp nautica2_next_step_ultra_local_seo.zip root@YOUR_SERVER_IP:/root/
```

On the server:

```bash
backend_dir=/websites/nautica/backend
mkdir -p "$backend_dir"
unzip /root/nautica2_next_step_ultra_local_seo.zip -d "$backend_dir"
```

If the ZIP extracts into a nested `nautica2/` directory, flatten it:

```bash
if [ -d "$backend_dir/nautica2" ]; then
  shopt -s dotglob
  mv "$backend_dir/nautica2"/* "$backend_dir/"
  rmdir "$backend_dir/nautica2"
fi
```

Fix ownership:

```bash
chown -R nautica:nautica "$backend_dir"
```

Create or update the `current` symlink only **after** the build succeeds.

## 7. Production environment file

Create `/etc/nautica/nautica.env`:

```env
# Frontend
# Leave blank when Nginx serves the frontend and proxies /api and /uploads on the same HTTPS origin.
VITE_API_BASE_URL=

# API
NODE_ENV=production
PORT=8787
FRONTEND_ORIGIN=https://your-domain.tld
DATABASE_PATH=/websites/nautica/shared/db/nautica.sqlite
MEDIA_UPLOAD_DIR=/websites/nautica/shared/uploads
MEDIA_MAX_FILE_SIZE_MB=8
SESSION_COOKIE_NAME=nautica.sid
SESSION_TTL_HOURS=24
ENFORCE_HTTPS=true
ADMIN_BOOTSTRAP_EMAIL=admin@your-domain.tld
ADMIN_BOOTSTRAP_PASSWORD=CHANGE_THIS_TO_A_LONG_RANDOM_PASSWORD
ADMIN_2FA_ENABLED=false
ADMIN_2FA_SECRET=
ADMIN_2FA_WINDOW=1
BUSINESS_TIMEZONE=Europe/Zagreb
SMTP_FROM=no-reply@your-domain.tld
NOTIFICATION_PROVIDER=console
FACEBOOK_PAGE_URL=https://www.facebook.com/nauticatribunj
INSTAGRAM_PROFILE_URL=https://www.instagram.com/nautica_tribunj/
INSTAGRAM_GRAPH_ACCESS_TOKEN=
INSTAGRAM_USER_ID=
INSTAGRAM_FEED_LIMIT=6
SOCIAL_FEED_CACHE_TTL_SECONDS=300
```

Secure the file:

```bash
chown root:nautica /etc/nautica/nautica.env
chmod 640 /etc/nautica/nautica.env
```

### Critical values

- `VITE_API_BASE_URL` should normally stay blank for this Nginx setup so the frontend calls `/api` and `/uploads` on the same origin
- `FRONTEND_ORIGIN` must exactly match the public HTTPS origin
- `DATABASE_PATH` must point to **persistent storage outside releases**
- `MEDIA_UPLOAD_DIR` must point to **persistent storage outside releases**
- keep `PORT=8787` and proxy to it from Nginx
- only set `ADMIN_2FA_ENABLED=true` after generating and storing a valid Base32 authenticator secret

## 8. Install dependencies and build the app

Run as the `nautica` user inside the release directory:

```bash
sudo -u nautica -H bash -lc '
  cd /websites/nautica/backend
  npm ci
'
```

Then build:

```bash
sudo -u nautica -H bash -lc '
  set -a
  source /etc/nautica/nautica.env
  set +a
  cd /websites/nautica/backend
  npm run build
'
```

### What `npm run build` does here

For this repo it performs:

- frontend build into `dist/`
- prerender of public routes into `dist/<route>/index.html`
- `robots.txt` and `sitemap.xml` generation into `dist/`
- canonical, hreflang, Open Graph and schema output using `VITE_SITE_URL` from the loaded environment
- API compile into `server-dist/`

After the build succeeds:

```bash
rsync -a --delete /websites/nautica/backend/dist/ /websites/nautica/frontend/
chown -R nautica:nautica /websites/nautica/frontend
```

## 9. Database initialization and seeding

### First deployment

Run controlled bootstrap manually once before starting the service:

```bash
sudo -u nautica -H bash -lc '
  set -a
  source /etc/nautica/nautica.env
  set +a
  cd /websites/nautica/backend
  npm run db:migrate
  npm run db:seed
'
```

### Important codebase behavior

The current API startup path also runs migrations and seeding automatically on boot.
That means:

- first deployment: manual `db:migrate` and `db:seed` is still recommended for controlled bootstrap
- later deployments: normally just build + restart the API service
- only run `npm run db:seed` manually again when you intentionally want seed upserts or reference-data refresh behavior

### Check that the database exists

```bash
ls -lah /websites/nautica/shared/db/
sqlite3 /websites/nautica/shared/db/nautica.sqlite ".tables"
```

## 10. systemd service for the API

Create `/etc/systemd/system/nautica-api.service`:

```ini
[Unit]
Description=Nautica API
After=network.target

[Service]
Type=simple
User=nautica
Group=nautica
WorkingDirectory=/websites/nautica/backend
EnvironmentFile=/etc/nautica/nautica.env
ExecStart=/usr/bin/npm run start:api
Restart=always
RestartSec=5
TimeoutStartSec=60
UMask=0027
NoNewPrivileges=true
PrivateTmp=true
ProtectSystem=strict
ProtectHome=true
ReadWritePaths=/websites/nautica/shared/db /websites/nautica/shared/uploads
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

Enable and start it:

```bash
systemctl daemon-reload
systemctl enable nautica-api
systemctl start nautica-api
systemctl status nautica-api --no-pager
journalctl -u nautica-api -n 100 --no-pager
```

### Health check

```bash
curl -I http://127.0.0.1:8787/health
curl -I http://127.0.0.1:8787/ready
```

Expected result: HTTP 200.

## 12. Nginx site configuration

Create `/etc/nginx/sites-available/nautica.conf`:

For a stricter production baseline, prefer the hardened example in [docs/examples/nginx-nautica.conf](/C:/Users/nikma/Desktop/nautica%20website/docs/examples/nginx-nautica.conf). It adds request throttling, tighter security headers, and cache rules for admin/API traffic.

```nginx
server {
    listen 80;
    listen [::]:80;
    server_name your-domain.tld www.your-domain.tld;

    location /.well-known/acme-challenge/ { root /var/www/html; }
    location / { return 301 https://$host$request_uri; }
}

server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    server_name your-domain.tld www.your-domain.tld;

    ssl_certificate     /etc/ssl/cloudflare/nautica-origin.pem;
    ssl_certificate_key /etc/ssl/cloudflare/nautica-origin.key;

    root /websites/nautica/frontend;
    index index.html;

    client_max_body_size 16m;

    # Static build assets
    location /assets/ {
        access_log off;
        expires 1y;
        add_header Cache-Control "public, immutable";
        try_files $uri =404;
    }

    location /icons/ {
        access_log off;
        expires 30d;
        add_header Cache-Control "public, max-age=2592000";
        try_files $uri =404;
    }

    location = /favicon.ico {
        access_log off;
        expires 30d;
        add_header Cache-Control "public, max-age=2592000";
        try_files $uri =404;
    }

    location = /site.webmanifest {
        access_log off;
        expires 1d;
        add_header Cache-Control "public, max-age=86400";
        try_files $uri =404;
    }

    location /site-media/ {
        access_log off;
        expires 30d;
        add_header Cache-Control "public, max-age=2592000";
        try_files $uri =404;
    }

    # Uploaded media stored in shared persistent storage
    location /uploads/ {
        alias /websites/nautica/shared/uploads/;
        access_log off;
        expires 7d;
        add_header Cache-Control "public, max-age=604800";
        try_files $uri =404;
    }

    # API
    location /api/ {
        proxy_pass http://127.0.0.1:8787;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
        proxy_set_header Connection "";
        proxy_connect_timeout 5s;
        proxy_send_timeout 60s;
        proxy_read_timeout 60s;
    }

    location = /health {
        proxy_pass http://127.0.0.1:8787/health;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
    }

    location = /ready {
        proxy_pass http://127.0.0.1:8787/ready;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
    }

    # Prerendered routes first, SPA fallback last
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

Enable the site:

```bash
ln -s /etc/nginx/sites-available/nautica.conf /etc/nginx/sites-enabled/nautica.conf
nginx -t
systemctl reload nginx
```

If the default site exists, remove it:

```bash
rm -f /etc/nginx/sites-enabled/default
nginx -t && systemctl reload nginx
```

## 13. Firewall

Do not expose the Node API directly to the internet.
Only allow SSH, HTTP, and HTTPS:

```bash
ufw allow OpenSSH
ufw allow 80/tcp
ufw allow 443/tcp
ufw enable
ufw status
```

Because the app currently listens on port `8787`, the firewall matters.
Nginx should be the only public entry point.

## 14. Cloudflare dashboard settings

### Required

1. Add the domain to Cloudflare.
2. Point `@` and `www` to the Ubuntu server IP.
3. Turn proxying **on** (orange cloud).
4. SSL/TLS mode: **Full (strict)**.
5. Install the matching Origin CA certificate on Nginx.

### Recommended after first successful deploy

- SSL/TLS -> Edge Certificates -> **Always Use HTTPS**: On
- SSL/TLS -> Automatic HTTPS Rewrites: On
- Caching: leave conservative at first; verify uploads and API are not cached incorrectly

### Optional hardening

After everything works, consider enabling **Authenticated Origin Pulls**.
Do not turn this on until the base site works correctly with Full (strict).

## 19. Troubleshooting

### 502 Bad Gateway

Usually means Nginx cannot reach the API.

Check:

```bash
systemctl status nautica-api --no-pager
journalctl -u nautica-api -n 200 --no-pager
curl -I http://127.0.0.1:8787/health
```

### 403 CSRF_MISMATCH in admin

Most common causes:

- frontend origin does not match `FRONTEND_ORIGIN`
- browser is not sending the admin cookie
- requests bypass Nginx/Cloudflare and hit a different origin directly
- stale session after deployment

Fix path:

- confirm `FRONTEND_ORIGIN=https://your-domain.tld`
- sign out and sign in again
- open the site only through the public HTTPS domain

### Uploaded images return 404

Most common causes:

- `MEDIA_UPLOAD_DIR` points somewhere else
- Nginx `location /uploads/` alias does not match the same directory
- permissions prevent Nginx from reading files

Verify:

```bash
echo "$MEDIA_UPLOAD_DIR"
ls -lah /websites/nautica/shared/uploads
namei -l /websites/nautica/shared/uploads
```

### Cloudflare 526 error

This means Full (strict) is enabled but the origin certificate is not valid for the hostname or is not installed correctly.

Check:

- certificate file path
- private key file path
- certificate hostname coverage
- Nginx reload success

### Redirect loop

Most common causes:

- Cloudflare SSL mode not set correctly
- origin or app forcing HTTPS in a conflicting way

For this stack, keep it simple:

- Nginx handles the HTTP -> HTTPS redirect
- Cloudflare uses **Full (strict)**

### SQLite permission or lock issues

Check ownership and permissions:

```bash
ls -lah /websites/nautica/shared/db
namei -l /websites/nautica/shared/db/nautica.sqlite
```

The `nautica` user must be able to write to the DB directory and the WAL/SHM files.

## 21. Recommended exact deployment order

If you want the safest first production deployment order, use this exact sequence:

1. prepare Ubuntu packages
2. install Node.js LTS
3. create user and shared directories
4. upload and unpack release
5. create `/etc/nautica/nautica.env`
6. install npm dependencies
7. build the app
8. create `current` symlink
9. run controlled `db:migrate` and `db:seed`
10. create and start `nautica-api.service`
11. install Cloudflare Origin CA cert on Nginx
12. create and enable the Nginx vhost
13. test locally with `curl`
14. enable Cloudflare proxy + Full (strict)
15. test public domain
16. log in to admin and do one real content update
17. take the first backup

That gives you the most controlled path with the fewest hidden surprises.
